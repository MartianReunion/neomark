# 块结构

Neomark 文档应当首先被切分为块的序列，然后每个块进行递归处理。本章节的目的是介绍 Neomark 的块结构。

## 字符的读取

Neomark 解析器首先会根据给定的编码读取给定的 Neomark 文档，它通常是 UTF-8 。随后， Neomark 解析器会对读取的字符进行清洗，具体的规则如下：

| 涉及到的段 | 替换为 | 描述 |
| --- | --- | --- |
| U+0000–0008 | U+FFFD | 无效的 C0 控制字符 |
| U+000E–001F | U+FFFD | 无效的 C0 控制字符 |
| U+0080–0090 | U+FFFD | 无效的 C1 控制字符 |
| U+0093-009F | U+FFFD | 无效的 C1 控制字符 |
| U+000B-000C | U+000A | 垂直制表符、换页符视为等同于换行 |
| U+0009 | U+0020 | 水平制表符的作用等同于空格 |

值得注意的是， C1 控制字符中U+0091和U+0092得到了保留，因为它们的作用类似于 Unicode 私有区。

## 行的读取

Neomark 解析器随后会进行对行的读取，Neomark 解析器支持 LF 或 CRLF 两种格式的换行符，而没有紧跟一个LF的CR不会被识别为换行；LF 或 CRLF 本身并不是行的一部分。这也与 Rust 标准库对 [`String::lines()`](https://doc.rust-lang.org/std/string/struct.String.html#method.lines)的实现一致。

每个行具有缩进量，缩进量等于该行的前导空格（U+0020）个数。尽管 Neomark 解析器被设计为可以处理任意空格数量作为一个缩进的情况（因为解析器只关注空格的相对数量），但在书写文档时建议用2个空格作为一个缩进。

如果一个行没有任何字符或只有空格，则该行被视为“空行”。

## 块的切分

Neomark 中有两类块，其中一类是文本自然书写形成的，称为“自然块”；而另一类是为了调用模板、函数来渲染嵌套内容的块，这类块叫作“调用块”。

空行和缩进量是切分块的重要依据，值得注意的是：

1. 空行用于分隔相邻的块。块和块之间的连续多个空行等价于单个空行。
2. 在讨论一个嵌套的调用块内部行的缩进时，需要仔细区分：如果正在将调用块作为一个整体讨论，那么不应当减去调用块头部的缩进量；相反，如果正在讨论调用块内部各个行对于调用块的作用，那么则应当逐级减去调用块头部的缩进。

### 自然块的切分

一个不是空行也不属于调用块的行一定是一个自然块的一部分。一个自然块应当从第一个应当属于自然块的行开始，直到遇到第一个空行或第一个调用块头部结束。空行并不是自然块的一部分。

自然块内行的缩进是被忽略的，意味着如果一个行有前导空格，这些空格不会被渲染。

### 调用块的切分

调用块的切分更为复杂，调用块一定从一个调用块头部开始，包含底部所有缩进严格大于该头部的行或空行——然而位于调用块末尾的空行并不是调用块的一部分。具体而言，位于内容行之间的空行被保留在调用块内部，但调用块最后一个内容行之后跟随的空行会被裁剪。无论该调用块内部有什么，它都可以被嵌套进这个调用块。如果一个调用块中嵌套有调用块，在后续步骤中应当被递归解析。

调用块头部是指调用块的首行，即忽略前导空格后、由两个英文冒号（`::`）引导的行。一个调用块可以只有头部，而没有内部行。

### 实例

这里有一份自我指涉的实例，用于说明这些语法。

```neomark
这里是第一个自然块。
这里仍然是第一个自然块。

但这里就是第二个自然块。


这里是第三个自然块，中间有两个空行，但也被忽略了。
::func
  这里是一个调用块的内部，即使调用块前没有空行，第三个自然块的解析也随之停止。
  ::func
    这是一个调用块内嵌套的调用块。

    这里仍然是嵌套的一部分，即使中间存在空行。
这里缩进同时不满足两个嵌套调用块的缩进，那么这里就是第四个自然块。

::func
一个调用块可以没有内部的行，而只有头部。

::func
  尽管 Neomark 解析器被设计为可以解析，但最好的写法仍然是在调用块的上下也留出空格。



此处为第五个自然块，注意到上一个调用块后有三个空行，
但它们都不是该调用块的一部分，因为它们出现在了调用块的末尾。
```

上述实例中：

- 第一个和第二个自然块由一个空行分隔，第三个自然块前有两个连续空行——它们与单个空行的分隔效果相同。
- 第一个 `::func` 调用块嵌套了另一个 `::func` 调用块。嵌套调用块内部有一个空行，它被保留在内层调用块中；外层的嵌套块会在后续步骤中被递归解析。
- 第二个 `::func` 没有缩进大于头部的内部行，因此它是一个只有头部的调用块。
- 第三个 `::func` 后的三个空行出现在调用块末尾，它们不属于该调用块，而是作为块分隔符。

## 调用块头部的解析

调用块头部遵循以下格式：

```neomark
::name parameter1=value1 "parameter 2"="value 2" bool_parameter ...
```

两个英文冒号后紧跟的是被调用对象的名称。如果名称本身包含空格，也可以用双引号包裹，例如`::"my name"`。冒号后紧跟名称是标准的写法，但介于有人习惯于在冒号后加空格， Neomark 解析器也被设计忽略冒号后的空格，例如：

```neomark
::  name
```
随后紧跟的是使用一个或多个空格隔开的参数，一个标准的参数包括参数名和参数的值，两者用等号隔开。如果参数名或参数值本身就包含空格，则需要用引号包裹。引号内部分支持转义：`\"`和`\\`。

上面例子中的`bool_parameter`是一种语法糖，它的作用等同于`bool_parameter=true`，即参数名单独出现时，将该参数的值设置为`true`。

此外，参数值的传递和存储均使用字符串，不会进行类型检查或转换。
